Monitoring vs Observability

The "What" vs the "Why"

Monitoring

  • Answers known-unknown
  • Predefined questions
  • Dashboards and threshold alerts

Observability

  • Explores unknown-unknowns
  • Arbitrary queries
  • Explains system behavior

Prerequisite to Observability

OpenTelemetry Ingestion (The Open Standard)

OpenTelemetry Ingestion Flow
  • Standard Protocol: OTLP unifies logs, metrics, and traces
  • Portable Pipeline: process, enrich, and sample in one neutral layer
  • Backend Flexibility: route to AWS and third-party tools without app rewrites
  • Future-Proofing: reduce lock-in risk and keep migration paths open

Prerequisite to Observability

Proprietary Ingestion (The Vendor-Locked Path)

Proprietary Ingestion Flow
  • Vendor-Specific Agents: telemetry collection tied to one platform
  • Closed Data Model: schema and query behavior coupled to vendor tooling
  • High Switching Cost: migrations often require re-instrumentation and pipeline changes
  • Reduced Optionality: less leverage on pricing, architecture, and roadmap control

Receiver Pattern

Receiver Diagram Placeholder
  • Enforce one canonical ingestion endpoint
  • Map signal sources by protocol
  • Keep app teams vendor-agnostic

Processor Pattern

Processor Diagram Placeholder
  • memory_limiter for load spikes
  • batch to reduce backend write pressure
  • Tail sample only error/slow transactions

Exporter Pattern

Exporter Diagram Placeholder
  • Mirror traffic during tool migrations
  • Route logs, metrics, traces independently
  • Preserve optional multi-home strategy

      - name: error-traces
        type: status_code
      - name: high-latency
        type: latency
        latency: { threshold_ms: 1200 }

Head vs Tail Sampling

Head Sampling

Tail Sampling

  • Captures all critical outliers
  • Needs buffering resources
